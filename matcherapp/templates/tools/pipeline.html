Pipeline Architecture

End-to-end retrieval and scoring layers

L0

Ingestion & Sanitization

Raw resumes (PDF, DOCX, TXT) are parsed by TextExtractor. Extracted text passes through a 7-detector adversarial sanitizer:

  • Prompt injection — catches instruction-override attempts hidden in text
  • Invisible/white text — detects keyword stuffing via zero-opacity characters
  • Homoglyphs — Unicode lookalike substitutions (е vs e)
  • JD duplication — copy-pasted job description text inside resume
  • Excessive keyword density — suspiciously high TF for target terms
  • Font-size manipulation — tiny text used to inflate keyword count
  • Non-resume gate (LLM) — gpt-4o-mini YES/NO classifier rejects cover letters, portfolios, etc.

Flagged documents receive a threat_level of HIGH and zero score without entering downstream stages.

L1

Hybrid Retrieval — BM25 + Dense Bi-Encoder

Two independent retrievers score each resume against the JD:

  • BM25 — lexical term overlap, rewards exact keyword matches, robust for technical skill terms
  • Dense bi-encoder (all-MiniLM-L6-v2) — semantic cosine similarity in 384-d embedding space

Both scores are normalized to [0,1] before fusion. This stage acts as a fast pre-filter.

L2

RRF Fusion

Reciprocal Rank Fusion combines BM25 and dense rankings:

RRF(d) = Σ 1 / (k + rank_i(d)) where k=60

RRF is parameter-light, handles rank ties gracefully, and consistently outperforms score-level weighted averaging when retriever calibrations differ.

L3

Cross-Encoder Re-Ranking

Top-N candidates from RRF pass through a cross-encoder (ms-marco-MiniLM-L-6-v2) which jointly encodes JD+resume and produces a relevance logit. Sigmoid-transformed to [0,1].

Cross-encoders are ~10× slower than bi-encoders but far more accurate on nuanced semantic matching — hence applied only to the shortlist, not the full corpus.

L4

LLM-as-Judge — 4-Dimension Scoring

An LLM (gpt-4o-mini, temp=0.0) evaluates each candidate on four structured dimensions, returning a validated JSON schema:

D1 — Hard Skills (×0.40)

Skill match coverage: BUILT_WITH / USED / LISTED / ABSENT per required skill

D2 — Experience Depth (×0.35)

Seniority alignment, years, scope of ownership vs JD requirements

D3 — Domain Fit (×0.15)

Industry, product type, and context transferability

D4 — Constraints (×0.10)

Location, visa, availability, certifications, hard requirements

Dim composite = D1×0.40 + D2×0.35 + D3×0.15 + D4×0.10. Final score = α×cross_encoder + (1−α)×dim_composite. Falls back to deterministic regex scoring if LLM unavailable.

OPT

ESCO Ontology — Skill Adjacency Graph

The ESCO occupation/skill graph allows partial credit for adjacent skills. A candidate missing "Kubernetes" but with "Docker Swarm" receives a penalized partial match through graph traversal rather than a zero.

Same-level skills in the ontology are matched with a configurable penalty (default 0.7×). This prevents hard misses on near-equivalent technologies and improves nDCG in the ablation study.

EVAL

Evaluation Metrics

Against golden labels (golden_dataset.json):

  • nDCG@k — Normalized Discounted Cumulative Gain at k=3,5,10
  • MRR — Mean Reciprocal Rank (first relevant result position)
  • Precision@k — Fraction of top-k results that are relevant
  • Spearman ρ — Rank correlation between predicted and golden order
