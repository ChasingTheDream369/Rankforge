Production Roadmap

Next steps to harden and scale the system

1

LLM + Skill Knowledge Graph

LLM as brain, ESCO graph as translator

Replace static regex skill extraction with a graph-grounded approach:

  • LLM extracts candidate skill mentions → normalised to ESCO concept URIs
  • Graph traversal finds same-level occupations and adjacent skills with configurable hop penalty
  • Scoring uses graph distance as partial credit rather than binary match
  • Enables "Python → NumPy → pandas → data wrangling" semantic chains without hard-coding synonyms

Effort: ~2 weeks. Requires ESCO SPARQL endpoint or local RDF graph (rdflib). Impact: highest — fixes false negatives on equivalent technologies.

2

Compliance & Bias Audit Layer

GDPR, EEOC, demographic parity

Every scoring run should produce an immutable audit log:

  • PII redaction before any LLM call (name, address, DOB, photo detection)
  • Per-dimension score distribution across protected attribute proxies
  • Disparate impact ratio (4/5ths rule) flagged automatically
  • Signed audit record stored separately from main DB for legal hold

Effort: ~1 week. extras/compliance.py is the starting point — needs DB persistence and per-run attachment.

3

MCP Server Integration

Tool calls for ATS / recruiter copilots

Expose the matching engine as an MCP-compatible tool server so recruiter AI assistants can call it directly:

  • match_resumes(jd, resumes[]) — returns ranked candidates with explanations
  • explain_score(resume_id) — deep-dive on a single candidate
  • update_label(resume_id, label) — human-in-the-loop feedback capture

extras/mcp_server.py has the skeleton — needs FastAPI wrapper and auth token validation.

4

Cost Tracker & Budget Controls

Per-run token accounting and hard limits

LLM costs accumulate fast at scale. Needed controls:

  • Token count logged per resume per stage (extraction, scoring, explanation)
  • Per-organisation monthly budget cap with soft warning + hard cutoff
  • Automatic fallback to deterministic scoring when budget exhausted
  • Cost breakdown surfaced in the dashboard per run

extras/cost_tracker.py has the accounting logic — needs DB model and per-run association.

5

Human-in-the-Loop Feedback

Recruiter labels feed back into golden dataset

Close the loop between recruiter decisions and model quality:

  • Thumbs up/down on candidate cards stored as relevance signals
  • Recruiter final decision (interviewed / hired / rejected) appended to golden dataset
  • Periodic re-evaluation of nDCG against accumulated labels surfaces model drift
  • Fine-tune LLM judge prompts or re-weight dimensions using feedback signal

extras/feedback.py is the starting point.

6

Parameter Tuning

Optimize α blend weight and dimension weights against golden labels

Current weights are set heuristically. With enough golden labels, tune via grid search or Bayesian optimization:

  • α (cross-encoder blend) — currently empirical; should be tuned per domain (tech vs. finance vs. healthcare)
  • D1–D4 dimension weights — 0.40/0.35/0.15/0.10 reflects general hiring priorities; role-type-specific weights will improve nDCG
  • RRF k constant — k=60 is the standard default; lower k rewards top-ranked results more aggressively
  • Run ablation study after each tuning iteration to track nDCG@3 and MRR on golden dataset

Effort: ~1 day once golden dataset has 50+ labelled pairs. The ablation page gives immediate feedback.

INF

Infrastructure Hardening

Replace in-process threads with proper task queue

  • Replace threading.Thread with Celery + Redis for durable background jobs
  • Postgres for all state (replace SQLite) — needed for multi-worker deployments
  • S3 / object storage for resume files instead of local FS
  • Rate limiting on API endpoints (django-ratelimit)
  • SSE or WebSocket for real-time progress instead of client polling
