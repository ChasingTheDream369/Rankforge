Production Roadmap

Next steps to harden and scale the system

1

LLM + Skill Knowledge Graph

LLM as brain, ESCO graph as translator

ESCO ontology adjacency is live for D1 and D3 scoring — missing skills trigger an LLM tool call when absent from the graph. Remaining work to complete the full knowledge-graph vision:

  • LLM extracts candidate skill mentions → normalised to ESCO concept URIs at extraction time (currently done ad-hoc via canonicalize_skill)
  • Multi-hop graph traversal beyond one level (e.g. "Python → NumPy → pandas → data wrangling" chains)
  • Configurable hop penalty schedule instead of a single 0.7× factor
  • Persist ESCO graph locally as RDF (rdflib) to remove SPARQL endpoint dependency

Effort: ~1 week. Impact: fixes false negatives on equivalent technologies across all domains.

2

Compliance & Bias Audit Layer

GDPR, EEOC, demographic parity

Every scoring run should produce an immutable audit log:

  • PII redaction before any LLM call (name, address, DOB, photo detection)
  • Per-dimension score distribution across protected attribute proxies
  • Disparate impact ratio (4/5ths rule) flagged automatically
  • Signed audit record stored separately from main DB for legal hold

Effort: ~1 week. extras/compliance.py is the starting point — needs DB persistence and per-run attachment.

3

MCP Server Integration

Tool calls for ATS / recruiter copilots

Expose the matching engine as an MCP-compatible tool server so recruiter AI assistants can call it directly:

  • match_resumes(jd, resumes[]) — returns ranked candidates with explanations
  • explain_score(resume_id) — deep-dive on a single candidate
  • update_label(resume_id, label) — human-in-the-loop feedback capture

extras/mcp_server.py has the skeleton — needs FastAPI wrapper and auth token validation.

4

Cost Tracker & Budget Controls

Per-run token accounting and hard limits

The two-stage LLM scorer (gpt-4o-mini extraction + gpt-4o scoring) and the agentic D2/verification passes accumulate cost quickly at scale. Needed controls:

  • Token count logged per resume per stage (extraction, scoring, D2 agent, verification retry)
  • Per-organisation monthly budget cap with soft warning + hard cutoff
  • Automatic fallback to deterministic scoring when budget exhausted (fallback path already exists)
  • Cost breakdown surfaced in the dashboard per run

extras/cost_tracker.py has the accounting logic — needs DB model and per-run association.

5

Human-in-the-Loop Feedback

Recruiter labels feed back into golden dataset

Close the loop between recruiter decisions and model quality:

  • Thumbs up/down on candidate cards stored as relevance signals
  • Recruiter final decision (interviewed / hired / rejected) appended to golden_dataset.jsonl
  • Periodic re-evaluation of nDCG against accumulated labels surfaces model drift
  • Fine-tune few-shot examples or re-weight dimensions using accumulated feedback

extras/feedback.py is the starting point.

6

Parameter Tuning

Optimize blend weight, few-shot examples, and role presets against golden labels

Role-adaptive dimension weight presets (Junior/Mid/Senior/Staff/Executive) and a custom weight override UI are already live. Remaining tuning work:

  • α (cross-encoder blend, currently 0.50) — tune per domain (tech vs. finance vs. healthcare) once golden labels exist per vertical
  • Role preset weights — presets are heuristic; grid search or Bayesian optimisation against per-role golden subsets will improve nDCG
  • Few-shot example selection — add Healthcare, Legal, and Product verticals to the domain-specific few-shot bank
  • RRF k constant (currently 60) — lower k rewards top-ranked results more aggressively; tune after retrieval-stage ablation
  • CE_TOP_PERCENT (currently 0.50) — increasing to 0.75 may improve recall at the cost of latency
  • Run ablation study after each tuning iteration to track nDCG@3 and MRR on the golden dataset

Effort: ~1 day per vertical once golden dataset has 50+ labelled pairs. The Ablation page gives immediate feedback.

INF

Infrastructure Hardening

Replace in-process threads with proper task queue

  • Replace threading.Thread with Celery + Redis for durable background jobs — currently a daemon thread fires per run with no retry on crash
  • Postgres for all state (replace SQLite) — required for multi-worker deployments and concurrent rescoring
  • S3 / object storage for resume files instead of local FS
  • Rate limiting on API endpoints (django-ratelimit) — especially start_run and rescore_run
  • SSE or WebSocket for real-time progress instead of client polling
